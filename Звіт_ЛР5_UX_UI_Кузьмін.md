### 2. Відповіді на питання

#### 2.1 Чим Task Flow відрізняється від User Flow?

**Task Flow** (потік задач) фокусується на конкретній задачі та її виконанні без прив'язки до конкретного користувача. Він показує послідовність кроків для досягнення певної мети. **User Flow** (потік користувача) є ширшим поняттям і включає весь шлях користувача від входу в систему до досягнення мети, враховуючи різні сценарії, точки прийняття рішень та можливі відгалуження.

#### 2.2 Чому вайрфрейми зазвичай виконуються у відтінках сірого?

Вайрфрейми виконуються у відтінках сірого з кількох причин:
- Дозволяє зосередитися на структурі, ієрархії та функціональності
- Допомагає уникнути суб'єктивних оцінок щодо дизайну на ранніх етапах
- Швидше створюються та легше змінюються
- Економить час та ресурси

#### 2.3 Що таке «Grey Box Method» у прототипуванні?

**Grey Box Method** (метод сірих прямокутників) - це техніка створення низькодеталізованих прототипів, де замість реальних елементів інтерфейсу використовуються прості сірі прямокутники різних розмірів. Кожен прямокутник представляє певний елемент інтерфейсу (зображення, текстовий блок, кнопка).

## Хід роботи

### Практичне завдання №1. Побудова логіки (User Flow / Task Flow)

На основі User Stories та CJM, створених у лабораторній роботі №4, була розроблена схема руху користувача для основного сценарію - додавання нового дедлайну в додаток **Deadline Tracker**.

**Схема включає:**
- Початкову та кінцеву точки (овали)
- Екрани додатку (прямокутники)
- Дії користувача (стрілки з підписами)
- Точки прийняття рішень (ромби): валідація даних, успішність збереження

**Основний сценарій:**
```
Головна сторінка → Натискання кнопки "+" → Екран додавання дедлайну → 
Введення даних → Натискання "Зберегти" → Валідація даних → 
Збереження на сервері → Екран успіху → Повернення на головну
```

**Альтернативні сценарії:**
- Якщо дані не валідні → Показати помилки → Повернення до форми
- Якщо помилка сервера → Показати повідомлення про помилку

### Практичне завдання №2. Створення Low-fidelity вайрфреймів

Було створено **6 екранів** додатку Deadline Tracker у форматі Low-fidelity Wireframes. Для створення використовувалися сірі прямокутники, лінії та плейсхолдери для зображень.

**Створені екрани:**
1. **Головна сторінка** - список дедлайнів з картками завдань, індикаторами пріоритету та кнопкою додавання
2. **Екран додавання дедлайну** - форма з полями для назви, дати, пріоритету та опису
3. **Екран успіху (Success State)** - підтвердження успішного додавання дедлайну
4. **Екран деталей** - перегляд повної інформації про дедлайн з кнопками редагування та видалення
5. **Календар** - візуальний перегляд дедлайнів у календарному форматі
6. **Нагадування** - список push-сповіщень та майбутніх нагадувань

**Використання Layout Grid:**
Для мобільного додатку використовувалася сітка з **4 колонками** з відступами margin 16-24px. Це забезпечує:
- Чітку структуру вирівнювання елементів
- Консистентні відступи між компонентами
- Простоту адаптації під різні розміри екранів
- Відповідність принципу 8-point system

### Практичне завдання №4. Створення Wireflow та інтерактивність

Було створено **Wireflow** - схему, що поєднує вайрфрейми стрілками, показуючи логіку переходів між екранами.

**Реалізовані переходи:**
- Головна → Новий дедлайн (тап по кнопці "+")
- Новий дедлайн → Успіх (збереження валідних даних)
- Успіх → Головна (повернення до списку)
- Головна → Деталі (тап по картці завдання)
- Деталі → Головна (кнопка "Назад")
- Головна → Календар (тап по іконці календаря)
- Головна → Нагадування (тап по іконці дзвінка)

**Стани інтерфейсу:**
- **Loading State** - індикатор завантаження при збереженні дедлайну
- **Error State** - повідомлення про помилку валідації або сервера
- **Success State** - підтвердження успішної операції
- **Empty State** - екран, коли немає дедлайнів

## Відповіді на контрольні запитання

### 1. Який елемент вайрфрейму вказує на те, що тут буде зображення?

У вайрфреймах для позначення місця, де буде розташоване зображення, використовується **плейсхолдер** у вигляді прямокутника з перехрещеними лініями (хрестиком) всередині. Цей елемент має пунктирну рамку або світло-сірий фон.

### 2. Навіщо використовувати реальні тексти замість "Lorem Ipsum" на етапі High-fi вайрфреймів?

Використання реальних текстів дозволяє:
- Оцінити реальну довжину тексту та його відображення
- Виявити проблеми з перенесенням слів та версткою
- Краще передати контекст використання інтерфейсу
- Полегшити тестування з користувачами
- Раніше виявити проблеми з локалізацією

### 3. Що таке "Happy Path" у сценарії користувача?

**Happy Path** («щасливий шлях») - це основний сценарій взаємодії користувача з продуктом, де все йде за планом без помилок, відхилень або виключень. Це ідеальний шлях від початку до досягнення мети.

### 4. Як Layout Grid допомагає у передачі макета розробнику?

Layout Grid допомагає розробнику:
- Забезпечує чіткі вказівки щодо розмірів та відступів
- Дозволяє легко переводити піксельні значення у відносні одиниці
- Стандартизує систему відступів (8-point grid)
- Забезпечує консистентність між екранами
- Полегшує комунікацію між дизайнером та розробником

### 5. Чому важливо проєктувати екран "Success State" (успіх), якщо дія відбувається на сервері?

Екран Success State є критично важливим для UX:
- Надає користувачеві чітке підтвердження успіху
- Запобігає повторному відправленню даних
- Створює позитивний емоційний відгук
- Може містити інформацію про наступні кроки
- Покращує доступність для користувачів з обмеженими можливостями
