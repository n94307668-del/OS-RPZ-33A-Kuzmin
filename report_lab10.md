## Завдання для попередньої підготовки

### 1. Яке призначення команди id?

Команда `id` відображає інформацію про поточного користувача: його UID (ідентифікатор користувача), GID (ідентифікатор основної групи), а також список усіх груп, до яких належить користувач. Це корисно для перевірки прав доступу та належності до груп перед виконанням операцій з файлами.

### 2. Як переглянути, які права доступу має власник файлу?

Для перегляду прав доступу до файлу використовується команда `ls -l <filename>`. Перші 10 символів рядка описують тип файлу та права доступу. Наприклад, `-rwxr-xr--` означає, що власник має права на читання, запис і виконання (rwx), група — на читання та виконання (r-x), а інші — тільки на читання (r--).

### 3. Як змінити власника групи?

Для зміни групи-власника файлу використовується команда `chown` або `chgrp`. Синтаксис: `chown :newgroup file` або `chgrp newgroup file`. Наприклад, `chgrp users file.txt` змінить групу-власника файлу file.txt на users. Для рекурсивної зміни групи в каталозі використовується опція `-R`: `chgrp -R users /path/to/dir`.

### 4. Як можна переглянути у терміналі, який тип поточного файлу? Наведіть приклади для різних типів файлів.

Для визначення типу файлу використовують команди `file` та `ls -l`:

* Звичайний файл: `ls -l` показує "-" на початку (наприклад, `-rw-r--r--`).
* Каталог: `ls -l` показує "d" (наприклад, `drwxr-xr-x`).
* Символічне посилання: `ls -l` показує "l" (наприклад, `lrwxrwxrwx`).
* Блочний пристрій: `ls -l` показує "b".
* Символьний пристрій: `ls -l` показує "c".
* Іменований канал (FIFO): `ls -l` показує "p".
* Сокет: `ls -l` показує "s".

### 5. Для чого використовуються дозволи Setuid та Setgid?

**Setuid (Set User ID)** — це спеціальний дозвіл для виконуваних файлів. Коли він встановлений, програма виконується з правами власника файлу, а не користувача, що її запустив. Наприклад, команда `passwd` має setuid, що дозволяє звичайним користувачам змінювати свій пароль у файлі `/etc/shadow`, який належить root.

**Setgid (Set Group ID)** має два застосування: для файлів — програма виконується з правами групи-власника; для каталогів — усі нові файли в цьому каталозі автоматично успадковують групу каталогу, а не основну групу користувача, що створює файл. Це корисно для спільних робочих каталогів.

### 6. Для чого в системі потрібен так званий "липкий біт" (Sticky Bit)? Наведіть приклади, коли цей дозвіл доцільно використовувати.

**Sticky bit** — це спеціальний дозвіл для каталогів. Коли він встановлений (позначається літерою `t` в правах доступу для інших, наприклад `drwxrwxrwt`), тільки власник файлу, власник каталогу або root можуть видаляти або перейменовувати файли в цьому каталозі.

**Класичний приклад** — каталог `/tmp` (`drwxrwxrwt`). Усі користувачі можуть створювати файли в `/tmp`, але не можуть видаляти чужі файли. Це важливо для безпеки в багатокористувацьких системах. Інший приклад — спільний каталог для завантаження файлів, де кожен користувач може додавати файли, але не може видаляти чужі.

## Відповіді на контрольні запитання

### 1. Наведіть приклади зміни прав доступу символічним методом (Symbolic Method).

Символічний метод використовує літери для позначення класів користувачів (u — user, g — group, o — others, a — all) та операторів (+ додати, - вилучити, = встановити) разом з правами (r, w, x).

**Приклади:**
* `chmod u+x script.sh` — додати власнику право на виконання.
* `chmod go-w file.txt` — забрати право на запис у групи та інших.
* `chmod a=r file.txt` — встановити для всіх тільки право на читання.
* `chmod u+rwx,g+rx,o-rwx file.txt` — власник отримує всі права, група — читання та виконання, інші — жодних.

### 2. Наведіть приклади зміни прав доступу числовим методом (numeric/octal method).

Числовий метод використовує вісімкові числа: 4 — читання (r), 2 — запис (w), 1 — виконання (x). Сума цих значень визначає права для кожного класу користувачів.

**Приклади:**
* `chmod 755 script.sh` — `rwxr-xr-x` (власник: все, група та інші: читання + виконання).
* `chmod 644 file.txt` — `rw-r--r--` (власник: читання + запис, група та інші: тільки читання).
* `chmod 700 private/` — `rwx------` (тільки власник має всі права).
* `chmod 777 shared/` — `rwxrwxrwx` (всі мають всі права — небезпечно!).
* `chmod 4755 prog` — `rwsr-xr-x` (setuid + стандартні права).

### 3. Яке призначення команди umask?

Команда `umask` (user file creation mode mask) встановлює маску створення файлів, яка визначає, які права за замовчуванням будуть вилучені при створенні нових файлів і каталогів. За замовчуванням повні права для файлів — 666, для каталогів — 777. Значення umask віднімається від цих базових значень.

Наприклад, якщо umask = 022, нові файли отримують права 644 (`rw-r--r--`), а каталоги — 755 (`rwxr-xr-x`). Команда `umask` без параметрів виводить поточне значення, а з параметром (наприклад, `umask 027`) змінює його.

### 4. Порівняйте жорсткі та символічні посилання.

| Критерій | Жорстке посилання (hard link) | Символічне посилання (soft/symbolic link) |
|---|---|---|
| Inode | Той самий inode, що й оригінал | Власний inode |
| Тип файлу | Звичайний файл (-) | Символічне посилання (l) |
| Посилання на каталог | Заборонено (крім root) | Дозволено |
| Межі ФС | Тільки в межах однієї ФС | Може перетинати межі ФС |
| Видалення цілі | Дані залишаються доступними | Посилання стає "битим" |
| Розмір | Розмір даних | Розмір шляху до цілі |
| Команда | `ln source target` | `ln -s source target` |

### 5. Чи можна виконати файл, для якого є права на виконання, але не встановлені права на читання (--x)? Поясніть.

Так, виконати можна, але з обмеженнями. Якщо файл є скриптом (bash, python тощо), інтерпретатору потрібно прочитати вміст файлу для виконання, тому без права на читання скрипт не запуститься. Однак якщо файл є скомпільованою бінарною програмою, вона може бути виконана безпосередньо операційною системою без права на читання, оскільки ядро завантажує бінарний код у пам'ять.

### 6. Якщо ми змінюємо права доступу та дозволи в поточній сесії, чи будуть вони збережені в наступній?

Так, зміни прав доступу до файлів та каталогів зберігаються у файловій системі та є постійними. Вони не залежать від сесії користувача і залишаться після виходу з системи, перезавантаження тощо. Однак зміни прав, внесені командами на кшталт `newgrp` або `umask`, діють лише в поточній сесії оболонки (shell).

### 7. Чи є якийсь шаблон, яким система користується щодо прав та доступів при створенні нових файлів? Як можна змінити права дозволу за замовчуванням?

Так, система використовує шаблон на основі значення `umask`. Базові права: файли — 666, каталоги — 777. Фактичні права = базові - umask. Наприклад, при umask 002 файли отримують 664, каталоги — 775.

Змінити права за замовчуванням можна командою `umask`. Наприклад, `umask 077` робить усі нові файли приватними (600 для файлів, 700 для каталогів). Для постійного застосування umask налаштовується у файлах профілю оболонки (`~/.bashrc`, `~/.profile`, `/etc/profile` тощо).

### 8. Яким чином можна створити жорстке посилання? В яких ситуаціях їх доцільно використовувати?

Жорстке посилання створюється командою `ln <оригінал> <посилання>` (без опцій). Жорстке посилання — це додаткове ім'я для існуючого файлу, яке вказує на той самий inode.

**Доцільно використовувати:**
* Коли потрібен альтернативний шлях до файлу без дублювання даних.
* Для резервного копіювання важливих файлів у різних каталогах.
* Коли видалення оригінального імені не повинно призводити до втрати даних (поки є хоча б одне посилання, файл існує).

**Обмеження:** жорсткі посилання не можуть вказувати на каталоги та не можуть перетинати межі файлових систем.

### 9. Яким чином можна створити символічне посилання? В яких ситуаціях їх доцільно використовувати?

Символічне (м'яке) посилання створюється командою `ln -s <ціль> <посилання>`. Це окремий файл, який містить шлях до цільового файлу чи каталогу.

**Доцільно використовувати:**
* Для створення ярликів на файли в зручних місцях.
* Для посилань на каталоги (наприклад, `/var/www/html -> /home/user/site`).
* Для посилань між різними файловими системами.
* Для версіонування бібліотек (наприклад, `lib.so -> lib.so.1.2.3`).

### 10. Уявіть, що програмі потрібно створити одноразовий тимчасовий файл, який більше ніколи не знадобиться після закриття програми. Який правильний каталог для створення цього файлу?

Для одноразових тимчасових файлів, які автоматично видаляються після перезавантаження, слід використовувати каталог `/tmp`. Для тимчасових файлів, які повинні зберігатися між перезавантаженнями, використовується `/var/tmp`. Альтернативно можна використовувати команду `mktemp`, яка створює унікальний тимчасовий файл у `/tmp` з безпечним випадковим іменем.

### 11. Є файл оригінал та для нього створено два посилання — символічне та жорстке. Що відбудеться з іншими файлами, якщо видалити: файл оригінал, символічне посилання, жорстке посилання?

* **Видалення оригінального файлу:** жорстке посилання продовжує працювати (дані не втрачені, оскільки inode все ще має лічильник посилань > 0). Символічне посилання стає "битим" (dangling link) — вказує на неіснуючий файл, при спробі доступу видає помилку "No such file or directory".
* **Видалення символічного посилання:** нічого критичного не відбувається. Видаляється тільки посилання, оригінальний файл та жорстке посилання залишаються незмінними.
* **Видалення жорсткого посилання:** зменшується лічильник посилань на inode. Якщо це було останнє посилання (разом з оригіналом), дані фізично видаляються. Якщо лишається ще хоча б одне посилання (оригінал), файл продовжує існувати.
